Making SysML v2 learnable through UX.

2026 · UX input for side project of Cauê Napier, Systems Engineer at Swedish Space Corporation

Problem

SysML v2 is powerful but hard to learn without a course first — and the spec hands every presentation and interaction choice to the tool.

Solution

I reframed a "review the screens" brief into a learnability foundation: ten design principles, a fixed-vs-flexible constraint map separating spec from tool, and an interface audit with a ranked roadmap.

Impact

Principles, audit, and a leverage-ranked roadmap — foundations that outlive any single screen, ordered to unblock the most downstream work. Ongoing.

Why this case matters

Design-engineering signal
  1. Reframed the scope. Brief: "review the screens." Outcome: principles, audit, roadmap. Principles outlive screens.
  2. Designed inside a specification. Separated what SysML v2 fixes from what tooling can change. Rare skill in B2B / enterprise.
  3. MBSE domain literacy. SysML v2, systems-of-systems, expert workflows. Not generic frontend depth.

Problem

Make SysML v2 modeling learnable without a course first. The spec (released September 2025) fixes element semantics and notation; presentation and interaction are left to each tool. So every decision sits on one side of a boundary — what the standard requires, or what the tool can change.

The standard fixes semantics. Everything else is the tool design space.

Design principles

Ten principles drawn from UX and information design literature, then reviewed with Cauê against how systems engineers actually work. Green = directly supports learnability.

Synthesized from
Tufte (data-ink, layering, small multiples) · Bertin (visual variables) · Nielsen (heuristics) · Shneiderman (overview + detail, linked views) · Norman (affordances, safe exploration) · Cognitive load (Sweller, Miller) · SysML v2 spec · domain review with a practicing systems engineer.
01
One model, many views

Single source of truth. Diagram, tree, table, text all sync. Shneiderman; linked views

02
Consistent visual grammar

Same meaning = same appearance. Enables pattern recognition. Bertin; Tufte

03
Progressive disclosure

Show what the task needs. Hide the rest. Miller's Law; NNGroup

04
Click anywhere, land everywhere

Selection syncs across all views instantly. Linked views

05
Context preservation

Never zoom in and get lost. Orientation supports learning. Shneiderman's mantra

06
Type-first organization

Group by element type, not creation order. IA

07
Edge density management

Filter or limit edges to reduce noise. Tufte: layering & separation

08
Validation timing

Real-time vs. on-demand. User controls interruption. B2B expert workflows

09
Safe exploration

Generous undo. Non-destructive defaults. Mistakes are cheap. Norman

10
Leverage familiar patterns

Build on IDE/CAD conventions users already know. Jakob's Law

SysML v2 constraint map

Where the spec constrains design, and where it doesn't. Shape and semantics are fixed. Presentation is where learnability happens.

Fixed

Set by SysML v2 specification

ElementStandard representation
Part def / Part usageRectangle with «part def» / «part» keyword
CompositionFilled diamond on parent end
Port / ConnectorSmall square on block edge / line between ports
RequirementRectangle with «requirement», id field
State / TransitionRounded rectangle / arrow with trigger label
Flexible

Open to design

AspectDesign freedom
ColorUnspecified. Can encode type, subsystem, status, health.
SizeCan encode information (node size = child count, criticality).
Layout & edge routingAuto-layout, spacing, alignment, orthogonal vs. curved, bundling.
Hierarchy displayContainment vs. separate diagrams.
Labels, filtering, animationDefaults, truncation, hover expansion; visibility; zoom transitions.
Why the boundary matters: Shape is fixed. Color, size, layout are presentation. Keeping the layers orthogonal — shape = type, color = status, size = magnitude — means each reads independently.

Concrete recommendations

Near-term moves, ordered by what unblocks the most downstream work.

  1. Resolve Model Explorer vs. File Explorer. Biggest IA decision. Gates sidebar hierarchy, icon weight, default view.
  2. v1 → v2 bridging in search. "Block" surfaces "Part" with the v1 label visible. High-leverage, small implementation.
  3. Standardize icon style and hover behavior. Mixed flat/filled icons weaken the visual grammar. One-pass cleanup before IA hardens.
  4. Map core workflows end-to-end. Create → add parts → connect → review/validate. Surfaces gaps static analysis misses.
  5. Tooltip patterns that teach SysML v2. Extend the Part tooltip systematically. Strongest lever for teaching through use.

Reflection

Limit I couldn't resolve: learnability is testable only with real users mid-task. Until then, every principle here is a hypothesis. The recommendations are ranked by leverage, not by evidence.

Most MBSE tools assume training. System Forge asks what happens if the interface itself teaches. Every tooltip, cue, and error message becomes a chance to help users learn — not just use — the language.
