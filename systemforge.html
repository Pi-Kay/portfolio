⬠ System Forge · Graph Modeling Learnability

Reducing cognitive load and enabling navigation in visual complexity — design principles for B2B systems engineering software

2025–26 · SysML2, MBSE · Learnability, Cognitive Load, Visual Navigation

Research Question

How can a systems modeling tool reduce cognitive load for engineers from any domain — enabling navigation through visual complexity without the usual training overhead, and without sacrificing the power to handle real-world scale?

Context

System Forge is a model-based systems engineering (MBSE) tool built for learnability across engineering domains. It integrates SysML2 concepts where they help, but prioritizes reducing cognitive load over strict specification compliance. The tool exists in early form — design principles do not yet exist.

SysML notation has two levels: semantics (what elements mean) and presentation (how they look). Semantic conventions must be followed. Presentation — layout, hierarchy, navigation patterns — is where learnability improvements can be made.

The Challenge

Draft design principles that balance three competing demands:

Familiarity
What engineers expect from modeling tools
Learnability
Low cognitive load without formal training
Capability
Navigation must scale to real complexity

The constraint: SysML2 was new territory. A SysML course assumes engineering background. UX bootcamp methods assume consumer apps with quick user testing. Systems engineers aren't available for hallway tests. This required a different approach to understanding learnability barriers.

Reframing the Problem

Early in the project, it became clear that jumping to UI elements was premature. The first task was defining principles — what guides decisions, not what pixels look like.

The realization: You can't design a dropdown or a panel without knowing what principles that dropdown serves. Principles first, then patterns, then components.

This reframing surfaced four interconnected challenges:

Challenge 1
B2B UX in Specialized Domains
Standard UX heuristics assume consumer contexts. Specialized B2B requires understanding what practitioners actually do — not what generic best practices suggest.
Challenge 2
SysML2 Constraints
What's fixed by the standard? What's open to interpretation? This boundary isn't documented — it had to be mapped explicitly.
Challenge 3
Integration Opportunities
Learning from other complex professional tools — CAD, IDEs, 3D modeling — where has UX successfully reduced cognitive load in dense interfaces?
Challenge 4
Relearning Cost
Where is improvement worth the switching cost? Reducing time to learn SysML2, reducing cognitive load, reducing errors — which changes justify relearning?

Beyond Surface-Level UX

A common expectation in B2B software: the designer handles the visual layer while engineers handle the domain. This works for surface fixes — but not for decisions that affect cognitive load and navigation.

Surface Fixes
  • Color contrast ratios
  • Keyboard shortcuts
  • Error message wording
  • Loading state indicators
Cognitive Load Decisions
  • Which data surfaces first
  • How views support navigation
  • What complexity to hide vs. expose
  • When to filter vs. when to zoom

What should be most prominent in the interface? That requires knowing what users do most often, what's done daily versus once a year, what mistakes increase cognitive load unnecessarily. None of that comes from UX heuristics alone.

Research Basis

The principles draw on cognitive research and information design literature — synthesized for application to learnability and navigation in systems modeling software.

Cognitive
Visual-Spatial Learners
Space engineers are disproportionately visual-spatial thinkers who need to see the whole picture before understanding parts. Semantic memory is structured as a network, not a list — graph-based navigation aligns with how engineers think (Collins & Loftus 1975; Silverman 1999).
Cognitive
Cognitive Load Theory
When information is represented inconsistently across views, cognitive load increases — decisions take longer and are substantially worse. Consistent navigation reduces the mental effort of cross-referencing (Sweller; PMC 2023).
Information Design
Bertin's Retinal Variables
Position, size, value (light/dark), color, shape — each supports different navigation tasks. Value is superior for encoding order; color hue for category. Every mark must reduce cognitive load, not add to it.
Information Design
Tufte's Principles
Data-ink ratio: every pixel should reduce cognitive load, not add visual noise. Small multiples: filtered sub-views enable navigation in dense graphs without losing orientation.

Where AI Helped

AI became genuinely useful — not as a replacement for expertise, but as a translation layer for understanding learnability barriers.

Translation
Plain Language Explanations
AI explains engineering concepts without assuming prior knowledge — reducing my own cognitive load while learning the domain. Metaphors, examples, historical context.
Synthesis
Framework Cross-Referencing
Checking if frameworks from cognitive science (load theory, visual variables) apply to systems modeling navigation. Fast connections to then validate with experts.
Mapping
Fixed vs. Flexible
SysML2 is a standard, but not everything is equally fixed. Semantic elements must be followed. Presentation — where learnability lives — can be improved.

The limits: AI generates synthesis that can fall apart under expert scrutiny. Quickly-generated content is quickly forgotten — it doesn't reduce long-term cognitive load. The skill is knowing what to ask AI and what to ask an expert.

Engineering vs. Design Thinking

Working closely with engineers surfaced a difference worth documenting: engineering and design are almost opposite cognitive modes.

Engineering Mode
  • Divide and conquer
  • Find a solution to this piece
  • Execute, move on
  • Sequential navigation
Design Mode
  • Connect parts
  • Consider all elements at once
  • Hold complexity without resolving prematurely
  • Simultaneous navigation

Neither mode is wrong. Engineering's precision produces reliable systems. Design's holistic view produces coherent experiences. The visual layer isn't separate from functionality — it emerges from understanding how navigation patterns connect all parts from the user's perspective.

User Progression Framework

Users at different stages have different cognitive load thresholds and navigation needs. The principles structure decisions around three user states:

Stage 1
New User
First encounter with the tool. Cognitive load must be minimal — clear guardrails, obvious affordances. Navigation must teach without tutorials.
Stage 2
Building User
Growing projects, increasing complexity. Needs visibility controls, multiple views. Navigation must support skill growth without overwhelming cognitive capacity.
Stage 3
Advanced User
Large-scale models, high density. Efficient navigation and density management. Cognitive load is acceptable because skill compensates — the tool must stay out of the way.

Deliverable: 10 Design Principles

Each principle addresses a specific source of cognitive load or navigation failure observed in existing MBSE toolchains. Organized by priority — what's architectural versus configurable versus optional.

Non-Negotiable
3 Principles
Remove any of these and cognitive load becomes unmanageable. Architectural decisions, not features.
Strong Default
3 Principles
Right navigation patterns for most teams. Should be configurable — a specific minority has legitimate reasons for alternatives.
Genuine Choice
2 Principles
Both options manage cognitive load differently. The right answer depends on team size and workflow maturity.
Optional
2 Principles
Genuinely good ideas for reducing load. Address in v2 — not blockers for a working v1.

Non-Negotiable

01 One Model, Many Views Non-Negotiable

A single source of truth with multiple synchronized representations. Diagram, tree, table, and text views all reflect the same underlying model. Changes in one view propagate immediately to all others.

Why non-negotiable: When views diverge, engineers must mentally reconcile differences — cognitive load spikes, errors multiply. Cross-referencing between unsynchronized documents is the single largest time sink in existing MBSE workflows. This is architectural: retrofitting synchronization is nearly impossible.

02 Consistent Visual Grammar Non-Negotiable

Same meaning = same appearance, everywhere. If a block is blue in the diagram, it's blue in the tree. If dashed lines mean "optional" in one context, they mean "optional" everywhere. No exceptions.

Why non-negotiable: Inconsistent visual encoding forces users to re-learn the interface in every view. Per Bertin's semiotics, visual variables (color, shape, line style) must map to semantic meaning consistently — otherwise they add noise rather than reduce load. This compounds: one inconsistency teaches users to distrust all visual cues.

03 Progressive Disclosure by Task Non-Negotiable

Show what the current task requires, hide everything else. Default views expose only the most common operations. Advanced features exist but don't clutter the primary interface. Complexity is available, never imposed.

Why non-negotiable: Cognitive load research (Sweller) shows working memory handles ~4 chunks. Exposing 50 options when 4 are needed doesn't feel "powerful" — it feels overwhelming. New users abandon tools that can't be learned incrementally. Progressive disclosure is how powerful tools become learnable tools.

Strong Default

04 Click Anywhere, Land Everywhere Strong Default

Selecting an element in any view highlights it in all views. Click a requirement in the tree → the diagram scrolls to show it → the properties panel updates → the table row highlights. Navigation is unified across representations.

Why strong default: Reduces the "where am I?" problem that plagues multi-view tools. Engineers working on complex models need to maintain orientation — synchronized selection is how. When to disable: During focused editing in a single view, constant synchronization can be disorienting. Advanced users may prefer manual sync to avoid view jumping during rapid edits.

05 Context Preservation on Zoom Strong Default

When zooming in or navigating to a detail, maintain visual context of where that detail sits in the larger structure. Breadcrumbs, minimap, or faded parent context — never "zoom in and get lost."

Why strong default: Visual-spatial thinkers (most engineers) need to see part-whole relationships to understand either. Losing context during zoom forces mental reconstruction of the whole — expensive and error-prone. When to disable: Maximum screen real estate may be needed for dense detail work. Experienced users with strong mental models may prefer full-screen detail views.

06 Type-First Organization Strong Default

Default organization groups elements by type (all requirements together, all blocks together) rather than by creation order or arbitrary structure. Type-based navigation answers "show me all X" instantly.

Why strong default: Engineers think in types: "I need to check all the interfaces" or "where are the requirements?" Creation-order organization serves no cognitive purpose — it's an implementation artifact. When to disable: Some workflows organize by subsystem or lifecycle phase. Teams with strong conventions may prefer custom hierarchies over type-based defaults.

Genuine Choice

07 Edge Density Management Genuine Choice

Two valid approaches to managing visual complexity from relationships: (A) Strict limits — cap visible edges per node, require manual expansion; or (B) Type-filtered display — show all edges but filter by relationship type on demand.

Why genuine choice: Both reduce cognitive load differently. Strict limits prevent overwhelm but may hide critical connections. Type-filtering preserves completeness but requires active filtering. Decision factors: Small models favor option B (see everything). Large models favor option A (prevent hairball). Team preference matters — some find hidden edges more stressful than dense displays.

08 Validation Timing Genuine Choice

Two valid approaches to model validation: (A) Real-time — errors surface immediately as red highlights, preventing invalid states; or (B) On-demand — validation runs when requested, allowing exploratory incomplete states.

Why genuine choice: Real-time validation catches errors early but interrupts flow during exploratory modeling. On-demand validation supports brainstorming but allows errors to accumulate. Decision factors: Early-stage design favors option B (sketch freely). Implementation-stage favors option A (catch mistakes immediately). Mixed workflows may need both modes.

Optional

09 Semantic Zoom Optional

Different levels of detail at different zoom levels. Zoomed out: blocks show only names. Zoomed in: blocks show attributes, ports, internal structure. The representation adapts to the viewing scale.

Why optional: Excellent for large models where detail-at-all-scales creates visual chaos. However, implementing well requires significant engineering effort and careful design of each zoom level. V2 consideration: Ship v1 with manual collapse/expand. Evaluate semantic zoom based on user feedback about scale pain points.

10 Collaborative Presence Optional

Show where other team members are working in real-time. Cursor positions, selection highlights, and "currently editing" indicators across the model. Reduce coordination overhead in team workflows.

Why optional: Valuable for distributed teams but adds technical complexity (real-time sync, conflict resolution) and visual noise (multiple cursors). V2 consideration: Focus v1 on single-user experience. Collaborative features require stable foundation — better to add presence to a working tool than debug sync issues while core UX is still forming.

Deliverable: SysML2 Constraint Map

SysML2 defines some things rigidly and leaves others open. This map documents the boundary — essential for knowing where learnability improvements can happen without breaking standard compliance.

Category Fixed by Standard Open for Improvement
Element Types Block, Part, Port, Requirement, Action, State, Constraint — these are defined. Can't invent new core types. Icons, colors, default sizes, grouping in palettes, naming conventions in UI.
Relationships Composition, reference, dependency, allocation, satisfy, derive — semantic meaning is fixed. Line styles beyond minimum requirements, routing algorithms, bundling, visibility filtering.
Notation Shapes Blocks are rectangles. Ports are small squares on edges. Requirements have specific compartments. Corner radius, shadow, padding, label placement, icon overlays, state indicators.
Compartments Which compartments exist (properties, operations, constraints) and what they contain. Collapse/expand behavior, default visibility, ordering within compartments, overflow handling.
Diagram Types BDD, IBD, Requirements, Parametric, etc. — diagram semantics are specified. How users create/switch diagrams, hybrid views, custom filtered views, dashboard layouts.
Textual Notation SysML v2 textual syntax is standardized (KerML-based). Must parse/export correctly. Syntax highlighting, autocomplete, error messages, formatting, inline documentation.
Model Structure Package hierarchy, namespaces, visibility rules (public/private), import/export. How hierarchy is visualized, navigation shortcuts, search/filter, favorites, recent items.
Stereotypes Stereotype mechanism exists. Standard stereotypes (when defined) have fixed meaning. Visual representation of stereotypes, custom stereotype styling, stereotype palettes.

The design space: Everything in the "Open" column is where System Forge can differentiate. The goal isn't to deviate from SysML2 — it's to make standard-compliant models easier to build, navigate, and understand than in any other tool.

Key Insight: Presentation vs. Semantics

SysML2's specification focuses on what elements mean, not how they look. A block is semantically a classifier with parts and ports — but whether it's rendered with rounded corners, drop shadows, or gradient fills is presentation. The standard constrains semantics; presentation is design space.

This creates opportunity. Existing tools often treat presentation as an afterthought — "we followed the spec, why isn't it usable?" The answer: the spec doesn't ensure usability. That's where design principles meet standard compliance.

Open Questions

Some decisions require user validation or are deferred to later phases:

  • Density thresholds: At what cognitive load point do filtering controls become mandatory vs. optional?
  • Color meaning: How much semantic weight can color carry before it increases rather than reduces cognitive load?
  • Cross-mode consistency: How much visual similarity between diagram/text/table before users expect identical navigation behavior?
  • Onboarding decay: When do teaching elements become cognitive clutter for experienced users?
  • Edge bundling: When does bundling help readability vs. obscure important relationships?
  • Default zoom: What's the right initial view — fit-to-content or standard scale?

These are documented as explicit open questions — flagged for validation, not ignored.

Reflection

On the work: The hardest part wasn't drafting principles — it was defining where SysML standards end and learnability decisions begin. That boundary isn't documented anywhere. Drawing the line is the work.

On learning unfamiliar domains: AI speeds up learning but the slower work — digesting, questioning, validating with practitioners — can't be skipped. Fast generation doesn't reduce long-term cognitive load. The skill is knowing when to accelerate and when to slow down.

On B2B UX: Most UX methods developed for consumer apps where cognitive load is managed through simplification. Specialized B2B is different: professional users accept higher load in exchange for capability. The goal isn't minimal load — it's appropriate load for the task and skill level.

On principles before pixels: The instinct is to start designing screens. But a dropdown without a principle is just a guess. Principles create constraints — and constraints make design decisions defensible, not arbitrary.
