⬠ System Forge · Graph Modeling Learnability

Reducing cognitive load and enabling navigation in visual complexity — design principles for B2B systems engineering software

2025–26 · SysML2, MBSE · Learnability, Cognitive Load, Visual Navigation

Research Question

How can a systems modeling tool reduce cognitive load for engineers from any domain — enabling navigation through visual complexity without the usual training overhead, and without sacrificing the power to handle real-world scale?

Context

System Forge is a model-based systems engineering (MBSE) tool built for learnability across engineering domains. It integrates SysML2 concepts where they help, but prioritizes reducing cognitive load over strict specification compliance. The tool exists in early form — design principles do not yet exist.

SysML notation has two levels: semantics (what elements mean) and presentation (how they look). Semantic conventions must be followed. Presentation — layout, hierarchy, navigation patterns — is where learnability improvements can be made.

The Challenge

Draft design principles that balance three competing demands:

Familiarity
What engineers expect from modeling tools
Learnability
Low cognitive load without formal training
Capability
Navigation must scale to real complexity

The constraint: SysML2 was new territory. A SysML course assumes engineering background. UX bootcamp methods assume consumer apps with quick user testing. Systems engineers aren't available for hallway tests. This required a different approach to understanding learnability barriers.

Beyond Surface-Level UX

A common expectation in B2B software: the designer handles the visual layer while engineers handle the domain. This works for surface fixes — but not for decisions that affect cognitive load and navigation.

Surface Fixes
  • Color contrast ratios
  • Keyboard shortcuts
  • Error message wording
  • Loading state indicators
Cognitive Load Decisions
  • Which data surfaces first
  • How views support navigation
  • What complexity to hide vs. expose
  • When to filter vs. when to zoom

What should be most prominent in the interface? That requires knowing what users do most often, what's done daily versus once a year, what mistakes increase cognitive load unnecessarily. None of that comes from UX heuristics alone.

Research Basis

The principles draw on cognitive research and information design literature — synthesized for application to learnability and navigation in systems modeling software.

Cognitive
Visual-Spatial Learners
Space engineers are disproportionately visual-spatial thinkers who need to see the whole picture before understanding parts. Semantic memory is structured as a network, not a list — graph-based navigation aligns with how engineers think (Collins & Loftus 1975; Silverman 1999).
Cognitive
Cognitive Load Theory
When information is represented inconsistently across views, cognitive load increases — decisions take longer and are substantially worse. Consistent navigation reduces the mental effort of cross-referencing (Sweller; PMC 2023).
Information Design
Bertin's Retinal Variables
Position, size, value (light/dark), color, shape — each supports different navigation tasks. Value is superior for encoding order; color hue for category. Every mark must reduce cognitive load, not add to it.
Information Design
Tufte's Principles
Data-ink ratio: every pixel should reduce cognitive load, not add visual noise. Small multiples: filtered sub-views enable navigation in dense graphs without losing orientation.

Where AI Helped

AI became genuinely useful — not as a replacement for expertise, but as a translation layer for understanding learnability barriers.

Translation
Plain Language Explanations
AI explains engineering concepts without assuming prior knowledge — reducing my own cognitive load while learning the domain. Metaphors, examples, historical context.
Synthesis
Framework Cross-Referencing
Checking if frameworks from cognitive science (load theory, visual variables) apply to systems modeling navigation. Fast connections to then validate with experts.
Mapping
Fixed vs. Flexible
SysML2 is a standard, but not everything is equally fixed. Semantic elements must be followed. Presentation — where learnability lives — can be improved.

The limits: AI generates synthesis that can fall apart under expert scrutiny. Quickly-generated content is quickly forgotten — it doesn't reduce long-term cognitive load. The skill is knowing what to ask AI and what to ask an expert.

Engineering vs. Design Thinking

Working closely with engineers surfaced a difference worth documenting: engineering and design are almost opposite cognitive modes.

Engineering Mode
  • Divide and conquer
  • Find a solution to this piece
  • Execute, move on
  • Sequential navigation
Design Mode
  • Connect parts
  • Consider all elements at once
  • Hold complexity without resolving prematurely
  • Simultaneous navigation

Neither mode is wrong. Engineering's precision produces reliable systems. Design's holistic view produces coherent experiences. The visual layer isn't separate from functionality — it emerges from understanding how navigation patterns connect all parts from the user's perspective.

Approach

Structured the work in three phases, cycling between domain research, framework synthesis, and expert validation.

Weeks 1–3
Foundations

Weeks 4–6
User Progression

Weeks 7–8
Synthesis

The fourth activity: Slower processing. Pencil and paper. Let it sink in. This can't be accelerated — and it's what makes the cognitive load research actually internalized, not just referenced.

User Progression Framework

Users at different stages have different cognitive load thresholds and navigation needs. The principles structure decisions around three user states:

Stage 1
New User
First encounter with the tool. Cognitive load must be minimal — clear guardrails, obvious affordances. Navigation must teach without tutorials.
Stage 2
Building User
Growing projects, increasing complexity. Needs visibility controls, multiple views. Navigation must support skill growth without overwhelming cognitive capacity.
Stage 3
Advanced User
Large-scale models, high density. Efficient navigation and density management. Cognitive load is acceptable because skill compensates — the tool must stay out of the way.

Deliverable: 10 Design Principles

The output: 10 principles organized by layer — model foundation, view design, interaction, and safety. Each principle addresses a specific source of cognitive load or navigation failure observed in existing MBSE toolchains.

Non-Negotiable
3 Principles
Remove any of these and cognitive load becomes unmanageable. Architectural decisions, not features.
Strong Default
3 Principles
Right navigation patterns for most teams. Should be configurable — a specific minority has legitimate reasons for alternatives.
Genuine Choice
2 Principles
Both options manage cognitive load differently. The right answer depends on team size and workflow maturity.
Optional
2 Principles
Genuinely good ideas for reducing load. Address in v2 — not blockers for a working v1.

Examples: "One model, many views" is non-negotiable — without it, cognitive load from cross-referencing becomes unmanageable. "Click anywhere, land everywhere" is a strong default for navigation — but automatic sync across all views can be disorienting during focused work. "Managing edge density" is a genuine choice between strict enforcement and type-filtered display — both reduce visual complexity differently.

Open Questions

Some decisions require user validation or are deferred to later phases:

  • Density thresholds: At what cognitive load point do filtering controls become mandatory vs. optional?
  • Color meaning: How much semantic weight can color carry before it increases rather than reduces cognitive load?
  • Cross-mode consistency: How much visual similarity between diagram/text/table before users expect identical navigation behavior?
  • Onboarding decay: When do teaching elements become cognitive clutter for experienced users?

These are documented as explicit open questions — flagged for validation, not ignored.

Reflection

On the work: The hardest part wasn't drafting principles — it was defining where SysML standards end and learnability decisions begin. That boundary isn't documented anywhere. Drawing the line is the work.

On learning unfamiliar domains: AI speeds up learning but the slower work — digesting, questioning, validating with practitioners — can't be skipped. Fast generation doesn't reduce long-term cognitive load. The skill is knowing when to accelerate and when to slow down.

On B2B UX: Most UX methods developed for consumer apps where cognitive load is managed through simplification. Specialized B2B is different: professional users accept higher load in exchange for capability. The goal isn't minimal load — it's appropriate load for the task and skill level.
