⬠ System Forge · Graph Modeling Learnability

Design principles for reducing cognitive load in B2B systems engineering software

2025–26 · SysML2, MBSE · Learnability, Cognitive Load, Visual Navigation

Context

System Forge is a model-based systems engineering (MBSE) tool in early development. The founder — Cauê Napier, systems engineer at the Swedish Space Corporation (SSC) — is building it as a side project with the goal of making SysML2-based modeling more accessible across engineering domains.

At this stage, the product is taking shape: core architecture, data model, initial interface. The opportunity: establish design principles early, before patterns solidify. The question: what should guide UX decisions in a tool where users need professional capability, not consumer simplicity?

Framing and Reframing the Problem

Initial framing: Support the product with UX — review screens, improve layouts, ensure usability basics are covered.

The reframe: Before reviewing any screen, define what principles that screen should follow. SysML2 specifies what elements mean and how they relate — it deliberately leaves presentation and interaction to tooling. That's not a gap in the standard; it's the design space. The work is mapping that space: where does the standard constrain choices, and where are decisions ours to make?

10 Design Principles for System Forge

Draft principles synthesized from Cauê's product vision, UX research, and B2B domain considerations. Sources noted — some require validation.

01 One Model, Many Views Product Vision

Single source of truth with synchronized representations. Diagram, tree, table, text all reflect the same model. Changes propagate immediately. Graph view is the exception: provides overview, not a clickable navigation mode.

02 Consistent Visual Grammar UX Research

Same meaning = same appearance, everywhere. If dashed lines mean "optional" in one context, they mean "optional" in all contexts. Ref: Bertin's visual variables.

03 Progressive Disclosure B2B UX

Show what the current task requires, hide everything else. Complexity is available, never imposed. Ref: Nielsen Norman Group, enterprise UX patterns.

04 Click Anywhere, Land Everywhere To Validate

Selecting an element in any view highlights it in all views. Note: May not apply to graph view. At minimum: clear indication of current location and how to switch modes.

05 Context Preservation on Zoom UX Research

When zooming to detail, maintain visual context of where that detail sits. Breadcrumbs, minimap, faded parent — never "zoom in and get lost."

06 Type-First Organization UX Research

Default organization groups elements by type (all requirements together) rather than creation order. Answers "show me all X" instantly.

07 Edge Density Management To Validate

Two approaches: (A) Strict limits — cap visible edges; or (B) Type-filtered display — show all, filter on demand. Decision needed on preferred approach.

08 Validation Timing Product Decision

Two approaches: (A) Real-time errors; or (B) On-demand validation for exploratory work. Ref: B2B UX research on expert workflows. Requires product decision.

09 Safe Exploration Learnability

Learners need to try things without risk. Generous undo, clear recovery paths, non-destructive defaults. Mistakes are cheap; hesitation is expensive.

10 Leverage Familiar Patterns Learnability

Build on what engineers already know — from IDEs, CAD tools, drawing applications. Novel interactions must justify their learning cost. Ref: Transfer of learning.

SysML2 Constraint Map

SysML2 standardizes semantics and core notation. Presentation and interaction — how users navigate, filter, and manipulate views — is explicitly left to tooling. This table maps that boundary for System Forge.

Category Specified by Standard Tool Design Space
Element Types Block, Part, Port, Requirement, Action, State — defined types Icons, colors, default sizes, palette grouping, UI naming
Relationships Composition, reference, dependency, allocation — semantic meaning fixed Line styles, routing algorithms, bundling, visibility filtering
Notation Blocks are rectangles, ports are small squares on edges Corner radius, shadow, padding, label placement, state indicators
Compartments Which compartments exist and what they contain Collapse/expand, default visibility, ordering, overflow handling
Diagrams BDD, IBD, Requirements, Parametric — diagram semantics specified Create/switch UX, hybrid views, filtered views, dashboard layouts
Textual SysML v2 textual syntax standardized (KerML-based) Syntax highlighting, autocomplete, error messages, inline docs
Structure Package hierarchy, namespaces, visibility rules Hierarchy visualization, navigation, search/filter, favorites

Design Language Foundations: Tufte

Core principles from Edward Tufte's information design work — applied to systems diagrams.

Data-Ink Ratio
Every visual element earns its place. If a line, border, or label doesn't carry meaning, remove it.
→ Decorative chrome obscures semantic weight
Small Multiples
Same visual grammar, repeated across instances. Consistency is what makes diagrams learnable.
→ This is what a design language is
Layering & Separation
Different information at different visual levels (color, weight, position, size) — readable independently or together.
→ Distinguishes semantic vs. presentational
Micro/Macro Reading
A well-designed diagram works at a glance and under close inspection.
→ Maps to large-scale model navigation
Escaping Flatland
Showing multi-dimensional relationships in 2D without confusion.
→ SysML has deep structure; diagrams are flat

Next Steps

With principles established, the work moves to application and validation.

01
Workflow Documentation
Map common tasks (setting up a BDD, creating requirements, linking elements) with principles applied. Each workflow becomes a test case: does the UI support the principle?
02
User Testing & Interviews
Validate principles with practitioners. Do the learnability assumptions hold? Where do real workflows diverge from expected patterns?
03
UI Component Library
Build components that embody principles. Each component documents which principles it implements and how — making the connection explicit and reviewable.

Future considerations: Semantic zoom (automatic level-of-detail at different scales) and collaborative presence (real-time multi-user awareness) — valuable but dependent on stable foundations.

Reflection

On the work: SysML2 defines semantics; presentation is the tool's responsibility. Translating that into concrete guidance — "you can style this element, but this shape is fixed" — required working through the spec systematically. That translation is the constraint map.

On principles before pixels: The expectation for UX Designers is often visual output. But a dropdown without a principle is just a guess. Principles create constraints — and constraints make design decisions defensible, not arbitrary. And why general principles can be applied independently. Applying some principles without a good understanding and overview of all relevant principles would result in uneven quality and individual fixes.
