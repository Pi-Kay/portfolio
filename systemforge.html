⬠ System Forge · Design Principles

Making SysML2 learnable through UX — not courses

2026 - today · Collaboration with Cauê Napier (Systems Engineer, SSC)

Context

System Forge is a model-based systems engineering (MBSE) tool in early development. Cauê Napier, systems engineer at the Swedish Space Corporation (SSC), is building it to make SysML2-based modeling more accessible — without requiring users to take a course first.

The core question: can good UX reduce the learning curve for a complex modeling language? Can design decisions make the difference between "I need training" and "I can figure this out"?

The Learnability Goal

Two distinct user journeys, one shared challenge: learning SysML2 through the tool itself.

Design for Learning

New to SysML

Engineers who need MBSE but haven't used SysML before. The tool should teach the language through use — progressive disclosure, clear feedback, safe exploration. No course required.

Transitioning v1 → v2

Experienced SysML users adapting to v2's new terminology and concepts. The tool should bridge old and new — showing familiar names alongside v2 equivalents, surfacing what changed and why.

Framing and Reframing

Initial framing: Support the product with UX — review screens, improve layouts.

The reframe: Before reviewing screens, define what principles those screens should follow — with learnability as the primary lens. SysML2 specifies semantics; presentation and interaction are left to tooling. That's where learning happens.

Design Principles for System Forge

Synthesized from product vision, UX research, and B2B considerations. Green = directly supports learnability.

01
One Model, Many Views

Single source of truth. Diagram, tree, table, text all sync. Ref: Tufte "Escaping Flatland"

02
Consistent Visual Grammar

Same meaning = same appearance. Consistency enables learning. Ref: Bertin; Tufte "Small Multiples"

03
Progressive Disclosure

Show what the task requires, hide the rest. Learn as you go. Ref: NNGroup; Miller's Law

04
Click Anywhere, Land Everywhere

Selection syncs across views. Ref: Linked views pattern

05
Context Preservation

Never "zoom in and get lost." Orientation supports learning. Ref: Tufte; Shneiderman

06
Type-First Organization

Group by type, not creation order. Ref: Information architecture

07
Edge Density Management

Limit or filter edges to reduce noise. Ref: Tufte "Layering"

08
Validation Timing

Real-time vs. on-demand error checking. Ref: B2B expert workflows

09
Safe Exploration

Generous undo, non-destructive defaults. Mistakes are cheap. Ref: Norman's DOET

10
Leverage Familiar Patterns

Build on IDE/CAD conventions. Reduce novelty cost. Ref: Jakob's Law

Reflection

On learnability as a design goal: Most MBSE tools assume training. System Forge asks: what if the interface itself teaches? Every tooltip, every visual cue, every error message is an opportunity to help users learn — not just use — the language.

On principles before pixels: The expectation for UX Designers is often visual output. But a dropdown without a principle is just a guess. Principles create constraints — and constraints make design decisions defensible, not arbitrary.
