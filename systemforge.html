⬠ System Forge · Visual Design Language

Establishing visual standards for a systems modeling tool — learnable without dumbing down

2025–26 · SysML v2, Systems Engineering · Visual Standards, Design Systems

Research Question

How can a systems modeling tool be visually accessible to engineers from any domain — without the usual training overhead, and without sacrificing the power to handle real complexity?

Context

System Forge is a new systems modeling tool built to be accessible to engineers across domains. It integrates SysML v2 concepts where they help, but prioritizes usability over strict specification compliance. The tool exists in early form — visual standards do not yet exist.

SysML notation has two levels: semantics (what elements mean) and presentation (how they look — layout, color, hierarchy, navigation, views). Semantic conventions must be followed. Presentation is where usability improvements can be made.

Why I took this on: Having worked at Valispace — where requirements management meets systems engineering — I understood how engineers interact with complex modeling tools. The gap between powerful tooling and actual usability is where most tools fail. This project addresses that gap at the visual layer.

The Challenge

Establish a visual language that balances three competing demands:

Familiarity
What engineers expect from modeling tools
Accessibility
Learnable without formal training
Capability
Must handle real complexity at scale

The constraint: SysML v2 provides the semantic foundation — what elements mean. Presentation decisions are where learnability and usability improvements happen. The work is finding where conventions can be improved without breaking meaning.

Approach

Structured the work in three phases, with each phase contributing to all three deliverables incrementally.

Weeks 1–3
Foundations
Core principles, understand tool direction, document borrowed vs. improved conventions
Weeks 4–6
User Progression
New user onboarding, building user patterns, advanced user scale
Weeks 7–8
Synthesis
Consolidate findings, structure deliverables, document open questions

Guiding questions:

  • What visual conventions should the tool follow?
  • What can be borrowed from SysML vs. improved?
  • What makes diagrams learnable without formal training?
  • How do diagrams relate to text and table views?

User Progression Framework

Users at different stages need different visual support. The framework structures visual decisions around three user states:

Stage 1
New User
First encounter with the tool. First diagram creation. Needs reduced fear, clear guardrails, obvious affordances. Visual language must teach without tutorials.
Stage 2
Building User
Growing projects, increasing complexity. Needs visibility controls, sorting, multiple views. Visual language must support skill growth without overwhelming.
Stage 3
Advanced User
Large-scale models, high density. Needs efficient navigation, density management, power tools. Visual language must stay out of the way.

Each stage has distinct visual patterns documented in the pattern library — what to show, what to hide, how to scale complexity.

Trade-offs & Decisions

Conventions
Borrowed vs. improved
SysML v2 visual conventions were analyzed for what to keep, what to modify, and what to invent. Semantic elements stay standard. Presentation elements — spacing, hierarchy, color coding — are where improvements happen.
Consistency
Diagram, text, table
System Forge has three viewing modes. Visual language must translate across all three — what stays constant (element identity, color meaning) and what adapts (layout, density, interaction).
Learnability
Visual teaching without tutorials
Onboarding patterns built into the visual system itself — progressive disclosure, contextual hints, sensible defaults. The interface teaches through use, not documentation.
Scale
Managing density at complexity
Large models can have hundreds of elements. Visual hierarchy, filtering, and navigation patterns designed specifically for high-density views without losing orientation.

Deliverables

Three outputs, built incrementally — each phase contributed to all three.

Standards
Visual Design Document
Recommendations and guidelines. What conventions to follow, where to improve, how to make decisions. The rules.
Reference
Pattern Library
Collected examples and reusable patterns. Organized by user stage and use case. The toolkit.
Framework
Decision Method
How to approach future visual decisions. A framework for evaluating new patterns and edge cases. The thinking.

Why three outputs: A document alone doesn't help a designer make decisions about edge cases. A pattern library without principles becomes a grab-bag. A method without examples is abstract. The three work together.

Open Questions

Some decisions require user validation or are deferred to later phases:

  • Density thresholds: At what point do filtering controls become mandatory vs. optional?
  • Color meaning: How much semantic weight can color carry before it becomes overloaded?
  • Cross-mode consistency: How much visual similarity between diagram/text/table before users expect identical behavior?
  • Onboarding decay: When do "teaching" visual elements become clutter for experienced users?

These are documented in the deliverables as explicit open questions — flagged for validation, not ignored.

Reflection

On the work: The hardest part wasn't designing visuals — it was defining where SysML standards end and presentation decisions begin. That boundary isn't documented anywhere. Drawing the line is the work.

On user progression: A visual design language isn't just about aesthetics. It's about encoding progressive skill levels into the interface itself. What a new user needs to see is different from what an advanced user needs — and the visual system has to handle both without switching modes.
