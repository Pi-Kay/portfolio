Context

System Forge is a model-based systems engineering (MBSE) tool in early development, created by Cauê Napier, a systems engineer at the Swedish Space Corporation. The goal: make SysML2 modeling more accessible — without requiring users to take a course first.

I contribute UX analysis and design documentation as a side project collaboration. My role: establish design foundations that can guide development decisions — principles, interface evaluation, and a roadmap for systematic improvement.

The Technical Challenge

SysML2 (released September 2025) specifies element semantics and core notation, but leaves presentation and interaction design to individual tools. This creates both constraint and opportunity: the interface must teach a complex modeling language while respecting a strict specification. Every design decision must distinguish between what the standard requires and where tooling can innovate.

Framing and Reframing

Initial framing: Support the product with UX — review screens, improve layouts.

The reframe: Before reviewing screens, define what principles those screens should follow — with learnability as the primary lens. SysML2 specifies semantics; presentation and interaction are left to tooling. That's where learning happens.

What changed: The work became scoped to design foundations, not screen fixes. That shift made the deliverables different — principles, an interface audit mapped to those principles, and a roadmap — and made them more durable. Principles outlive individual screens.

The standard fixes semantics — shapes and notation carry meaning. Everything else is the tool design space, and that's where the work happens.

Two Audiences, One Interface

The tool must serve two distinct learning paths simultaneously — and the interface itself has to do the teaching.

Designing for Learnability

New to SysML

Engineers who need MBSE but haven't used SysML before. The tool should teach the language through use — progressive disclosure, clear feedback, safe exploration.

Transitioning v1 → v2

Experienced SysML users adapting to v2's new terminology. The tool should bridge old and new — showing familiar names alongside v2 equivalents.

Approach: Principles Before Pixels

For a new product with a solo designer, establishing principles early prevents arbitrary decisions later. Each phase feeds the next — principles constrain analysis, analysis constrains architecture, architecture constrains the design system.

Design Principles

Ten principles synthesized from product vision, UX research, and systems engineering requirements. Several principles emerged directly from Cauê's domain expertise — for instance, why Model Explorer visibility matters more than generic sidebar balance. Green = directly supports learnability.

01
One Model, Many Views

Single source of truth. Diagram, tree, table, text all sync. Ref: Shneiderman; linked-views pattern

02
Consistent Visual Grammar

Same meaning = same appearance. Consistency enables pattern recognition. Ref: Bertin; Tufte "Small Multiples"

03
Progressive Disclosure

Show what the task requires, hide the rest. Reduces cognitive load. Ref: Miller's Law; NNGroup

04
Click Anywhere, Land Everywhere

Selection syncs across all views instantly. Ref: Linked views pattern

05
Context Preservation

Never "zoom in and get lost." Orientation supports learning. Ref: Shneiderman's mantra

06
Type-First Organization

Group by element type, not creation order. Ref: Information architecture

07
Edge Density Management

Filter or limit edges to reduce visual noise. Ref: Tufte "Layering and Separation"

08
Validation Timing

Real-time vs. on-demand error checking — user controls interruption. Ref: B2B expert workflows

09
Safe Exploration

Generous undo, non-destructive defaults. Mistakes are cheap. Ref: Norman, Design of Everyday Things

10
Leverage Familiar Patterns

Build on IDE/CAD conventions users already know. Ref: Jakob's Law

SysML2 Constraint Map

Understanding where the specification constrains design vs. where tooling can innovate. The SysML2 specification defines semantic conventions that must be followed exactly — element shapes, relationship notation, diagram types. Everything else is presentation, and presentation is where learnability improvements happen.

Fixed

Must follow SysML2 specification

These elements carry semantic meaning in the standard. Changing their representation would break interoperability and confuse engineers familiar with the specification.

Element Standard Representation
Part def Rectangle with «part def» keyword
Part usage Rectangle with «part» keyword, shows type
Composition Filled diamond on parent end
Port Small square on block edge
Connector Line between ports
Requirement Rectangle with «requirement», id field
State Rounded rectangle
Transition Arrow between states with trigger label
Initial state Filled circle
End state Bullseye (circle in ring)
Flexible

Open to design — presentation choices

These aspects are not specified by SysML2. They are where tooling differentiates itself and where learnability improvements happen — without changing the meaning of the model.

Aspect Design Freedom
Color Not specified. Can assign by element type, subsystem, status, health.
Size Can encode information (node size = child count, criticality).
Layout Auto-layout algorithm is design choice. Spacing, alignment flexible.
Hierarchy display How to show nesting. Containment vs. separate diagrams.
Edge routing Orthogonal vs. curved. Bundling. Crossing minimization.
Labels What to show by default. Truncation. Hover expansion.
Filtering Which elements visible by default. Filter UI design.
Animation Transitions between states. Zoom behavior.
The boundary matters: Shape is fixed by SysML2 semantics — it cannot be redesigned. Color, size, and layout are presentation choices. Keeping these layers orthogonal (shape = type, color = status, size = magnitude) means each can be read independently without visual conflict.

Interface Analysis: Annotated Examples

The analysis evaluated first impressions and key workflow phases. Below are sample observations with annotations — noting both strengths and areas to explore further.

Example 1: Main Interface Layout
✓ Clear layout structure

Users can identify where to find tools and where diagrams appear. Icon groupings in left sidebar are visible and logically separated by function.

Supports: Jakob's Law
✓ Restrained color scheme

Interface is more modern than most systems engineering tools on first impression. Minimal visual clutter supports focus on the model content.

Supports: Professional trust
→ Sidebar icon differentiation

All sidebar icons have similar visual weight. Model Explorer could be more visually dominant since systems engineering workflows start from the model tree.

Principle: #06 Type-First
→ Icon style consistency

Some icons are flat line style while others have fills. Standardizing would strengthen visual grammar across the interface.

Principle: #02 Consistent Grammar
Example 2: Add Element Palette
✓ Grouped by category

Elements organized into meaningful categories (Structure, Behavior, Requirements, Connections, Organization, Documentation) rather than flat lists.

Supports: #06 Type-First
✓ Tooltips present

Hovering "Part" shows "Create an instance of a part" — helping users understand SysML2 concepts in context. This directly supports learnability.

Supports: #03 Progressive Disclosure
✓ Search available

"Search element types..." input allows direct access without navigating categories. Supports both novice (browse) and expert (search) patterns.

Supports: Flexibility
→ v1 terminology in search

When users search "Block" (v1 term), consider showing "Part" (v2 equivalent) with the old name visible. Supports users transitioning from SysML v1.

Reflection

On working with domain expertise: I'm not a systems engineer, and most of the principles that ended up on this page came from questions Cauê answered — why Model Explorer matters more than File Explorer, why element shapes are non-negotiable, why v1 users need a terminology bridge. UX for a specialized tool isn't applied from the outside; it's assembled through repeated checking against how the users actually work.

On documentation as design work: In early-stage products, clear documentation is as valuable as screens. Principles and analysis create shared understanding, reducing ambiguity and rework. The analysis report and roadmap are not byproducts — they're the foundation that makes future design decisions faster and more defensible.

On learnability as a design goal: Most MBSE tools assume training. System Forge asks: what if the interface itself teaches? Every tooltip, every visual cue, every error message is an opportunity to help users learn — not just use — a complex modeling language.
